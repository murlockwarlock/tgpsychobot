# PsychoNewBot

Платформа для Telegram-ботов с диалогами, управляемыми промптами, админ-панелью, подписками, тестами, контентом и отдельной реализацией для мессенджера MAX.

Один экземпляр приложения обычно обслуживает одного Telegram-бота и его базу данных. Поведение — системный промпт, темы, контент, тарифы, тесты и модели — настраивается из админки и хранится в БД.

## Возможности

- диалоги с контекстом, темами и настройками памяти;
- выбор AI-провайдера и модели из админки;
- текстовые, голосовые и визуальные сценарии: транскрибация, анализ изображений, генерация и редактирование изображений;
- база знаний с индексированием документов и поиском релевантного контекста;
- тесты с кнопками, свободными ответами, формулами и отдельным промптом результата;
- извлечение скрытых метаданных из ответа модели: блок `[DATA]...[/DATA]` сохраняется у пользователя и не показывается ему;
- контент, темы, медиа, рассылки, сегментация и экспорт пользователей/диалогов/метаданных;
- тарифы, промокоды, реферальная программа и платежи через Telegram Payments, YooKassa и Robokassa;
- companion-бот для MAX с общей бизнес-БД и отдельным хранением медиа платформы.

## Как устроено

```text
Пользователь Telegram / MAX
            │
            ▼
  Telegram Bot API / MAX Platform API
            │
            ▼
 handlers.py / max_messenger_bot
            │
    ┌───────┼────────────────┐
    ▼       ▼                ▼
 Диалоги  Тесты          Админка/рассылки
    │       │                │
    └───────┴──────► PostgreSQL / SQLite
                         │
                         ▼
       AI-провайдеры, платежи, файловая база знаний
```

Основной Telegram-бот запускается из `main.py`. В нём создаются aiohttp-приложение и aiogram dispatcher, подключаются обработчики, поднимаются фоновые задачи и регистрируются платёжные webhook-обработчики.

## AI-провайдеры

Провайдер, ключи и модели хранятся в конфигурации AI в БД и меняются через админку. Поддерживаются:

- **OpenAI** — чат, транскрибация, анализ изображений и работа с изображениями;
- **Google Gemini** — чат, мультимодальные сценарии, транскрибация и изображения;
- **Anthropic Claude** — чат и анализ изображений;
- **DeepSeek** — текстовый чат;
- **KIE** — чат, мультимодальные запросы, транскрибация, генерация и редактирование изображений.

У конфигурации может быть резервный провайдер/модель. Голосовой ответ сначала превращается в текст, затем проходит через тот же сценарий диалога, что и набранное пользователем сообщение.

### Скрытые метаданные пользователя

Промпт любого диалога или тестового анализатора может добавить в ответ блок:

```text
[DATA]
{"profile": {"interests": ["музыка"]}, "confidence": "high"}
[/DATA]
```

Блок вырезается до отправки пользователю, JSON объединяется с сохранёнными метаданными пользователя. В админке данные можно просматривать и экспортировать отдельно от истории сообщений.

## Telegram: Bot API, webhook и polling

Бот всегда использует **Telegram Bot API**: через него он отправляет сообщения, кнопки и медиа. Разница только в том, как Telegram доставляет входящие обновления.

- `webhook` — Telegram отправляет обновления HTTP POST-запросом на публичный HTTPS-адрес приложения. Это основной режим для продакшена.
- `polling` — приложение само периодически получает обновления методом `getUpdates`. Удобно локально.
- `auto` — сначала пытается настроить webhook, а при ошибке переходит на polling.

Режим задаётся переменной `TELEGRAM_DELIVERY_MODE`: `webhook`, `polling` или `auto` (по умолчанию). Один токен нельзя одновременно обслуживать polling-процессом и webhook-процессом.

Путь Telegram webhook строится так:

```text
BASE_WEBHOOK_URL + WEBHOOK_PATH_PREFIX + /webhook
```

Например: `https://bots.example.com/futuro/webhook`.

## Настройка окружения

Основной Telegram-процесс читает переменные из окружения; локально можно положить их в `.env`.

```dotenv
# обязательно
BOT_TOKEN=123456:telegram-token
DATABASE_URL=postgresql+asyncpg://user:password@localhost:5432/psychobot
OWNER_IDS=123456789,987654321
SERVER_IP=127.0.0.1
BASE_WEBHOOK_URL=https://bots.example.com

# HTTP и доставка обновлений
APP_PORT=8080
WEBHOOK_PATH_PREFIX=/my-bot
TELEGRAM_DELIVERY_MODE=auto
WEBHOOK_SETUP_RETRIES=10
WEBHOOK_RETRY_DELAY_SEC=15
POLLING_TIMEOUT=10
```

Для локального запуска без публичного HTTPS обычно достаточно указать `TELEGRAM_DELIVERY_MODE=polling`. `BASE_WEBHOOK_URL` и `SERVER_IP` всё равно проверяются при старте текущего приложения, поэтому для локального окружения можно использовать технические значения, например `http://localhost` и `127.0.0.1`.

`config.ini.example` оставлен как пример старого формата и используется как fallback в MAX-части. Для основного Telegram-бота источником конфигурации являются переменные окружения / `.env`.

## Локальный запуск

1. Создайте и активируйте виртуальное окружение Python 3.12+.
2. Установите зависимости проекта: aiogram, aiohttp, SQLAlchemy async, драйвер БД (`asyncpg` или `aiosqlite`), APScheduler, python-dotenv и SDK подключаемых провайдеров.
3. Создайте `.env` по примеру выше.
4. Запустите приложение:

```bash
python main.py
```

При первом старте `init_db()` создаёт таблицы и применяет безопасные добавочные изменения схемы. Для пустой SQLite-базы можно использовать URL вида:

```dotenv
DATABASE_URL=sqlite+aiosqlite:///./psychobot.db
```

## Тесты и формулы

Файл теста загружается из админки. Поддерживаются `.xlsx`, `.csv` и `.txt`; вопросы и формулы можно положить в один файл, а формулы — на лист `formulas` / `формулы`.

Для обычного вопроса:

```text
Вариант N  → текст ответа (и подпись кнопки, если «Кнопка N» пустая)
Кнопка N   → короткая подпись кнопки для длинного варианта
Значение N → число, передаваемое в формулу
```

Таким образом можно показать пользователю расшифровку вариантов в комментарии, дать кнопки `1 | 2 | 3`, сохранить полный текст ответа и отдельно использовать числовые баллы в формуле. Подробности и рабочий пример: [docs/universal_test_setup.md](docs/universal_test_setup.md).

Запуск проверок:

```bash
pytest -q
```

## Админка

Через Telegram-админку можно управлять:

- системным промптом, темами, кнопками, контентом и медиа;
- AI-провайдерами, моделями, лимитами и режимами памяти;
- базой знаний и индексированием файлов;
- тестами, формулами и отдельным промптом результата;
- пользователями, историей диалогов и скрытыми метаданными;

- тарифами, промокодами, реферальной программой и платежами.

## MAX

MAX — не прокси к Telegram, а отдельное приложение в `max_messenger_bot/`:

```bash
python -m max_messenger_bot.app
```

Оно использует общую БД для пользователей, промптов, тем, планов и бизнес-правил, но хранит состояния и медиа MAX отдельно. Telegram `file_id` нельзя использовать в MAX, поэтому медиа для MAX загружаются заново. Детали, переменные окружения и запуск: [max_messenger_bot/README.md](max_messenger_bot/README.md).

При развитии Telegram-функций следует отдельно оценивать перенос в MAX: общие модели и тестовая логика часто переиспользуются, но транспорт, кнопки, webhook и медиа отличаются.

## Фоновые задачи и платежные webhook

После запуска бот:

- обрабатывает очередь индексации базы знаний и очередь рассылок;
- каждые 15 минут проверяет подписки и баланс KIE;
- принимает callback-запросы YooKassa и Robokassa через aiohttp;
- записывает платёжные события в ротируемые логи `logs/payment_events_<port>.log`.

Для платёжных webhook внешний URL должен быть доступен провайдеру. Проверка платежа на стороне приложения выполняется через API провайдера и подписи, а не только по телу входящего запроса.

## Продакшен

В продакшене процесс обычно работает под PM2 через `ecosystem.config.js`; запускать его вручную поверх действующего процесса не нужно. Перед перезапуском конкретного бота проверьте его конфигурацию, порт, URL webhook и статус процесса.
